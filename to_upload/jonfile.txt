<div class="answercell post-layout--right">
    <div class="post-text" itemprop="text">
<p>TCP's primary design goal is to allow reliable data communication in the face of packet loss, packet reordering, and &mdash; key, here &mdash; packet duplication.</p>

<p>It's fairly obvious how a TCP/IP network stack deals with all this while the connection is up, but there's an edge case that happens just after the connection closes. What happens if a packet sent right at the end of the conversation is duplicated and delayed, such that <a href="https://en.wikipedia.org/wiki/Transmission_Control_Protocol#Connection_termination" rel="noreferrer">the 4-way shutdown</a> packets get to the receiver before the delayed packet? The stack dutifully closes down its connection. Then later, the delayed duplicate packet shows up. What should the stack do?</p>

<p>More importantly, what should it do if the program that owned that connection immediately dies, then another starts up wanting the same IP address and TCP port number?</p>

<p>There are a couple of choices:</p>

<ol>
<li><p>Disallow reuse of that IP/port combo for at least 2 times the maximum time a packet could be in flight. In TCP, this is usually called the 2&times;<a href="https://en.wikipedia.org/wiki/Maximum_Segment_Lifetime" rel="noreferrer">MSL</a> delay. You sometimes also see 2&times;<a href="https://en.wikipedia.org/wiki/Round_trip_time" rel="noreferrer">RTT</a>, which is roughly equivalent.</p>

<p>This is the default behavior of all common TCP/IP stacks. 2&times;MSL is typically between 30 and 120 seconds. (This is the <a href="https://en.wikipedia.org/wiki/Transmission_Control_Protocol#Protocol_operation" rel="noreferrer"><code>TIME_WAIT</code></a> period.) After that time, the stack assumes that any rogue packets have been dropped <em>en route</em> due to expired <a href="https://en.wikipedia.org/wiki/Time_to_live" rel="noreferrer">TTLs</a>, so it leaves the <code>TIME_WAIT</code> state, allowing that IP/port combo to be reused.</p></li>
<li><p>Allow the new program to re-bind to that IP/port combo. In stacks with <a href="https://en.wikipedia.org/wiki/Bsd_sockets" rel="noreferrer">BSD sockets</a> interfaces &mdash; essentially all Unixes and Unix-like systems, plus Windows via <a href="https://en.wikipedia.org/wiki/Winsock" rel="noreferrer">Winsock</a> &mdash; you have to ask for this behavior by setting the <code>SO_REUSEADDR</code> option via <a href="http://linux.die.net/man/2/setsockopt" rel="noreferrer"><code>setsockopt()</code></a> before you call <a href="http://linux.die.net/man/2/bind" rel="noreferrer"><code>bind()</code></a>.</p></li>
</ol>

<p><code>SO_REUSEADDR</code> is most commonly set in server programs.</p>

<p>The reason is, a common pattern is that you change a server configuration file and need to restart that server to make it reload its configuration. Without <code>SO_REUSEADDR</code>, the <code>bind()</code> call in the restarted program's new instance will fail if there were connections open to the previous instance when you killed it. Those connections will hold the TCP port in the <code>TIME_WAIT</code> state for 30-120 seconds, so you fall into case 1 above.</p>

<p>The safe thing to do is wait out the <code>TIME_WAIT</code> period, but in practice this isn't a big enough risk that it's worth doing that.  It's better to get the server back up immediately so as to not miss any more incoming connections than necessary.</p>

<div class="answercell post-layout--right">
    <div class="post-text" itemprop="text">
<blockquote>
  <p>SO_REUSEADDR allows your server to
  bind to an address which is in a<br>
  TIME_WAIT state.</p>
</blockquote>

<p>This socket option tells the kernel that even if this port is busy (in the TIME_WAIT state), go ahead and reuse it anyway. If it is busy, but with another state, you will still get an address already in use error.  It is useful if your server has been shut down, and then restarted right away while sockets are still active on its port.</p>

<p>From <a href="http://www.unixguide.net/network/socketfaq/4.11.shtml" rel="noreferrer">unixguide.net</a></p>

<div class="answercell post-layout--right">
    <div class="post-text" itemprop="text">
<p>When you create a socket, you don't really own it. The OS (TCP stack) creates it for you and gives you a handle (file descriptor) to access it. When your socket is closed, it take time for the OS to "fully close it" while it goes through several states. As EJP mentioned in the comments, the longest delay is usually from the TIME_WAIT state. This extra delay is required to handle edge cases at the very end of the termination sequence and make sure the last termination acknowledgement either got through or had the other side reset itself because of a timeout. <a href="http://developerweb.net/viewtopic.php?id=2941" rel="noreferrer">Here you can find</a> some extra considerations about this state. The main considerations are pointed out as follow : </p>

<blockquote>
  <p>Remember that TCP guarantees all data transmitted will be delivered,
  if at all possible. When you close a socket, the server goes into a
  TIME_WAIT state, just to be really really sure that all the data has
  gone through. When a socket is closed, both sides agree by sending
  messages to each other that they will send no more data. This, it
  seemed to me was good enough, and after the handshaking is done, the
  socket should be closed. The problem is two-fold. First, there is no
  way to be sure that the last ack was communicated successfully.
  Second, there may be "wandering duplicates" left on the net that must
  be dealt with if they are delivered.</p>
</blockquote>

<p>If you try to create multiple sockets with the same ip:port pair really quick, you get the "address already in use" error because the earlier socket will not have been fully released. Using SO_REUSEADDR will get rid of this error as it will override checks for any previous instance.</p>
